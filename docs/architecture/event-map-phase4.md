# Phase 4 — Domain Event Map (POS / Payments / Shifts / Offline)

- **Status:** PLAN / contract doc — **no code**. Purpose: lock event **payload shapes + required IDs now**, explicitly **shaped for the Phase-5 Accounting consumer** (the Phase-5 Codex review proved that POS events must carry COGS/tax/tender detail + stable reversal IDs, or the GL cannot post). Same discipline as `event-map-phase2.md`/`event-map-phase3.md`. This is the top Phase-4 readiness blocker (Task C) — it must be locked before POS implementation.
- **Transport:** transactional outbox (charter §24) — every event row written in the SAME tenant tx as the mutation (`emitEvent`). `occurredAt` is **server-injected** (last in the spread; device clocks untrusted, §14). Money = integer `*Minor` + `currency` + `scale`; quantities are **base-unit int8** (Decision #3).
- **Envelope (every event):** `id, type, version (default 1), tenant_id, correlation_id, request_id, payload (jsonb), status (pending), created_at`. Tenant-scoped, correlation/request-ID aware, versioned, replay-safe. **Version/additive-field/upcasting/replay policy: see `event-versioning.md`** (additive fields don't bump `version`; breaking changes bump + add a consumer upcaster).
- **`DomainEventType` constants to ADD (Phase 4):** `sale.refunded`, `sale.voided`, `payment.received`, `shift.opened`, `shift.closed`, `stored_value.issued`, `stored_value.redeemed`. **`sale.created` already exists but is MINIMAL** (`{ saleId, number, totalMinor, currency }` — verified) and **must be EXTENDED** to the shape below. Do NOT enumerate an event nothing emits (Phase-2 lesson).

> **Consumer-idempotency contract (Phase-5 INV-P5-7, CRITICAL).** The outbox can REDELIVER. The accounting consumer dedups on **`(tenant_id, outbox_event_id, posting_kind)`** (UNIQUE), posting in the same tx as the journal; a replay returns the existing journal, never re-posts. Therefore every event below carries the data needed to post **deterministically from the event alone** (no re-reading mutable OLTP state at post time).

> **Ordering/parking contract (Phase-5 INV-P5-8).** Reversal events (`sale.refunded`/`sale.voided`) carry `originalSaleId`; if the consumer sees a reversal before its original sale is posted, it **parks** the event until the dependency posts. Hence the stable source/original IDs on every event.

> **Reserved-field discipline (Phase-2 lesson):** any field a later consumer will bind but Phase 4 can't yet populate is reserved **present-but-null**, never shipped absent. Money fields are `_minor`-suffixed; carry `currency` + `scale`.

> **Functional-currency contract (Phase-5 INV-P5-4 — consumer-completeness gate, 2026-06-24).** The GL keeps its books in ONE **functional currency** (tenant/company); every journal line must balance **in functional currency** (INV-P5-1) and the consumer must post **without re-reading a mutable FX-rate table** (INV-P5-7). The `tenders[]` carry only `fxRateToSale` (tender→sale) — **NOT** the sale→functional rate the books need; a redelivered event re-reading a since-corrected rate table would post a different functional amount, breaking deterministic replay. Therefore **every financial event RESERVES (nullable) a functional-currency context** — `functionalCurrency`, `functionalScale`, and the **server-stamped `fxRateToFunctional`** (document-currency→functional, captured in the SAME tx at transaction time — same discipline as server-injected `occurredAt` and the `costing_method_applied` stamp; **never re-resolved later**) — **plus a pre-computed `*FunctionalMinor` twin beside every posting amount**, so the journal balances deterministically to the cent without the consumer re-dividing (multi-leg rounding would otherwise force a plug). **Reserved-nullable today; populated when the Phase-5 functional-currency model is locked AND the tenant enables multi-currency.** Single-currency tenants (the Guyana zero-config default): functional == transaction, rate = identity, twins == the `*Minor` amounts — **zero setup burden**. **Sequencing constraint (`cross-phase-dependencies.md`):** a historical FX rate cannot be backfilled, so a tenant must not run **multi-currency** POS in production until the producer stamps `fxRateToFunctional` (which needs only a `functional_currency` setting + spot-rate capture — a small seam, NOT the full Phase-5 revaluation model). Single-currency POS is unaffected. **`sale.voided` is EXEMPT** — it parks on and reverses the original posted journal, carrying no amounts.

## Event catalog

### `sale.created` — EXTEND (exists, minimal today)
- **Producer:** `pos.createSale` router.
- **Phase-4 consumer:** receipt; sales read model; audit. **Future:** **P5 Accounting** (revenue, tax-payable, inventory-asset↓/COGS, cash-clearing), P12 Analytics.
- **Payload (extended):** `{ saleId, number, locationId, companyId, shiftId, salesRepId, customerId, saleType: "sale", exchangeGroupId, currency, scale, functionalCurrency, functionalScale, fxRateToFunctional, subtotalMinor, discountMinor, taxMinor, totalMinor, subtotalFunctionalMinor, discountFunctionalMinor, taxFunctionalMinor, totalFunctionalMinor, lines:[{ saleLineId, skuId, productId, lotId, qtyBase, qtyScale, unitPriceMinor, lineDiscountMinor, lineTaxMinor, taxRateId, cogsMinor, cogsCurrency, cogsScale, cogsFunctionalMinor, costingMethodApplied }], taxBreakdown:[{ taxRateId, rate, baseMinor, taxMinor, baseFunctionalMinor, taxFunctionalMinor }], tenders:[{ tenderId, method, tenderCurrency, tenderScale, amountMinor, fxRateToSale, settledAmountMinor, settledFunctionalMinor }], offline:{ deviceId, terminalId, monotonicCounter, localTs, payloadVersion }, createdBy, occurredAt }`. **`exchangeGroupId` reserved nullable** — set (shared with the paired `sale.refunded`) only when this sale is the outbound leg of an exchange; null otherwise (re-gate MEDIUM). **Functional-currency fields (`functionalCurrency`, `functionalScale`, `fxRateToFunctional`, all `*FunctionalMinor` twins) reserved-nullable** per the functional-currency contract above — `fxRateToFunctional` is sale-currency→functional; `cogsFunctionalMinor` carries the COGS basis (possibly a different currency, per `cogsCurrency`/`cogsScale`) pre-converted to functional; `settledFunctionalMinor` is tender→functional **direct** (avoids tender→sale→functional double-rounding). The FX gain/loss balancing line is derived by P5 from `totalFunctionalMinor` vs Σ`settledFunctionalMinor` (consumer-completeness gate).
- **Required IDs:** saleId, locationId, companyId, per-line saleLineId + skuId, per-tender tenderId. `shiftId`/`salesRepId`/`customerId` reserved nullable (no shift/rep/customer captured ⇒ null). **`cogsMinor` per line drives the inventory-asset↓/COGS posting — populated because Phase 4 closes #8 (INV-P4-3 runs `applyValuation`); it is a FULL money triple (`cogsMinor` + `cogsCurrency` + `cogsScale`, since the inventory cost basis can differ in currency/scale from the sale — ADR-0008/INV-P5-4); `costingMethodApplied` stamps which method valued it.** `taxBreakdown` drives VAT-payable per rate. `tenders[]` drives cash-clearing per tender (FX recorded). `lotId` nullable.
- **GL posting it drives (P5):** Dr cash-clearing/tender, Cr revenue (subtotal−discount), Cr VAT payable (per rate); Dr COGS, Cr inventory-asset (per line `cogsMinor`).

### `sale.refunded` — NEW
- **Producer:** `pos.refund` router (first-class return, `saleType="return"`, links `originalSaleId`).
- **Future consumer:** **P5 Accounting** (reverse revenue/tax/COGS proportionally), commission clawback (INV-P4-7/Decision #7), P12.
- **Payload:** `{ saleId, originalSaleId, number, locationId, companyId, shiftId, currency, scale, functionalCurrency, functionalScale, fxRateToFunctional, saleType: "return", exchangeGroupId, subtotalMinor, discountMinor, taxMinor, totalMinor, subtotalFunctionalMinor, discountFunctionalMinor, taxFunctionalMinor, totalFunctionalMinor, lines:[{ saleLineId, originalSaleLineId, skuId, productId, lotId, qtyBase, unitPriceMinor, lineTaxMinor, taxRateId, restockedValueMinor, cogsCurrency, cogsScale, restockedValueFunctionalMinor, costingMethodApplied, restockLocationId }], taxBreakdown:[{ taxRateId, rate, baseMinor, taxMinor, baseFunctionalMinor, taxFunctionalMinor }], tenders:[{ tenderId, method, tenderCurrency, tenderScale, amountMinor, fxRateToSale, settledAmountMinor, settledFunctionalMinor }], refundReason, refundedBy, occurredAt }`. **`exchangeGroupId` reserved nullable** — set (shared with the paired `sale.created`) only when this return is the inbound leg of an exchange; null otherwise (re-gate MEDIUM). **Functional-currency fields reserved-nullable** per the contract above; the reversal must reverse **exactly** the original posting's functional amounts — to keep replay deterministic and symmetric, the refund's `fxRateToFunctional` / `*FunctionalMinor` are taken from the **original sale's stamped rate** (via `originalSaleId`/`originalSaleLineId`), NOT re-quoted at refund time (a re-quote would post a spurious FX gain/loss on a pure reversal).
- **Required IDs:** saleId (the return doc), **`originalSaleId` (ordering/parking key)**, per-line `originalSaleLineId`. **`restockedValueMinor` (full triple with `cogsCurrency`/`cogsScale`) = the value RESTOCKED to inventory.** ⚠️ **Producer note (Codex gate HIGH-4):** `applyValuation()` returns `cogsMinor: 0` on a positive/restock movement (COGS is computed on *issue*, not receipt), so the producer CANNOT take this from the costing primitive on the restock leg. The Phase-4 return path **derives `restockedValueMinor` from the original sale line's posted COGS basis** (referenced via `originalSaleLineId` — the original `sale.created` stamped per-line `cogsMinor`+`costingMethodApplied`), so the refund reverses **exactly** what the sale posted (symmetric inventory-asset/COGS reversal). `costingMethodApplied` carries the original method. `restockLocationId` reserved nullable (damaged-return ⇒ may not restock — then `restockedValueMinor=0`). `taxBreakdown` is the locked shape (matches `sale.created`) for per-rate VAT reversal. `tenders[]` carry full FX (match sale/payment) for realized-FX on the refund tender. Partial refunds carry only the refunded lines/qty.
- **GL posting it drives (P5):** reverse the proportional sale postings (revenue/VAT per `taxBreakdown`); restock inventory-asset at `restockedValueMinor`, reversing COGS.

### `sale.voided` — NEW
- **Producer:** `pos.voidSale` router (whole-sale void; manager step-up, §22).
- **Future consumer:** **P5 Accounting** (full reversal if the sale was already posted), commission 100% clawback, P12.
- **Payload:** `{ saleId, originalSaleId, number, locationId, companyId, shiftId, currency, scale, totalMinor, voidReason, voidedBy, occurredAt }`.
- **Required IDs:** saleId, **`originalSaleId`** (= the voided sale's id; ordering key). A void is a full reversal — the consumer reverses the original sale's journal (or no-ops if never posted). Distinct from refund (partial, restock-aware).

### `payment.received` — NEW
- **Producer:** `pos` payment path (a tender settling against a sale; also standalone AR receipts later).
- **Future consumer:** **P5 Accounting** (cash/bank clearing → AR/revenue settlement), shift cash reconciliation, P12.
- **Payload:** `{ paymentId, saleId, shiftId, locationId, companyId, tenderId, method, tenderCurrency, tenderScale, amountMinor, fxRateToSale, settledAmountMinor, saleCurrency, saleScale, functionalCurrency, functionalScale, tenderFxRateToFunctional, amountFunctionalMinor, settledFunctionalMinor, changeMinor, changeFunctionalMinor, realizedFxGainLossFunctionalMinor, receivedBy, occurredAt }`.
- **Required IDs:** paymentId, tenderId, saleId (nullable only for non-sale receipts), shiftId. `fxRateToSale`/`settledAmountMinor` carry the FX so the GL posts realized FX at settlement (P5). `changeMinor` for cash. **Functional-currency fields reserved-nullable** per the contract above: `tenderFxRateToFunctional` (tender→functional, server-stamped) + the `*FunctionalMinor` twins let P5 post Dr cash/bank, Cr clearing/AR, and the **realized FX gain/loss** (`realizedFxGainLossFunctionalMinor`) in functional currency without an FX-table re-read.
- **GL posting it drives (P5):** Dr cash/bank, Cr cash-clearing (or AR); realized FX gain/loss line where `fxRateToSale` differs.

### `shift.opened` — NEW
- **Producer:** `pos.openShift` router.
- **Future consumer:** shift/X-Z read model; **P5** (cash-drawer float as a cash account movement); P12.
- **Payload:** `{ shiftId, terminalId, locationId, companyId, cashierUserId, functionalCurrency, functionalScale, openingFloat:[{ currency, scale, amountMinor, fxRateToFunctional, functionalAmountMinor }], openedAt, openedBy, occurredAt }`.
- **Required IDs:** shiftId, terminalId, locationId, cashierUserId. `openingFloat[]` is multi-currency (split drawers, §12) — so the functional twin (`fxRateToFunctional`, `functionalAmountMinor`) is **per float row**, reserved-nullable per the contract above (a single envelope rate can't cover a multi-currency drawer).

### `shift.closed` — NEW
- **Producer:** `pos.closeShift` router (**blind close** — counted entered without seeing expected; system computes over/short).
- **Future consumer:** Z-report read model; **P5** (cash-drawer settlement to clearing; over/short to a shrinkage account); manager audit (§19/§22); P12.
- **Payload:** `{ shiftId, terminalId, locationId, companyId, cashierUserId, functionalCurrency, functionalScale, countedCash:[{ currency, scale, amountMinor, fxRateToFunctional, functionalAmountMinor }], expectedCash:[{ currency, scale, amountMinor, fxRateToFunctional, functionalAmountMinor }], overShort:[{ currency, scale, amountMinor, fxRateToFunctional, functionalAmountMinor }], cashMovements:[{ type, currency, scale, amountMinor, fxRateToFunctional, functionalAmountMinor }], zReportId, closedAt, closedBy, occurredAt }`.
- **Required IDs:** shiftId, zReportId. `expectedCash`/`overShort` are **system-computed** (blind close — the cashier never saw `expectedCash`); `overShort` drives the shrinkage posting + the manager audit signal. `cashMovements[]` = pay-ins/pay-outs/drops during the shift. **Per-row functional twins (`fxRateToFunctional`, `functionalAmountMinor`) reserved-nullable** per the contract above — every drawer/over-short/movement row is independently multi-currency, so the cash-settlement, shrinkage, and movement journals post in functional currency without an FX-table re-read.

### `stored_value.issued` — NEW (gift card / store credit)
- **Producer:** POS path that issues a gift card or store credit (sale of a gift card, or store credit granted on a return). Writes the stored-value balance ledger (INV-P4-5).
- **Future consumer:** **P5 Accounting** (recognize a **LIABILITY**, not revenue, on issue — charter §19), P12.
- **Payload:** `{ storedValueId, storedValueAccountId, movementId, kind: "gift_card" | "store_credit", saleId, paymentId, tenderId, customerId, amountMinor, currency, scale, functionalCurrency, functionalScale, fxRateToFunctional, amountFunctionalMinor, issuedBy, occurredAt }`.
- **Required IDs:** storedValueId, storedValueAccountId, movementId (the balance-ledger entry — the dedup/source key). `saleId`/`paymentId`/`tenderId` reserved nullable — a gift card SOLD at POS ties the issue-liability to the exact tender/payment clearing leg (present); a manual/goodwill store credit issues outside a sale (null). **Mirrors `stored_value.redeemed`'s `paymentId`/`tenderId` so issue and redeem are symmetric** (re-gate HIGH). `customerId` reserved nullable. **Functional twin (`functionalCurrency`/`functionalScale`/`fxRateToFunctional`/`amountFunctionalMinor`) reserved-nullable** so the liability posts in functional currency without an FX re-read. (The Dr account is resolved from a **stable, versioned account-mapping by `kind`** + nullable `saleId`/`paymentId` — an acceptable immutable-config lookup, not a mutable-OLTP re-read; gate LOW.)
- **GL posting it drives (P5):** Dr cash/clearing (gift-card sold) or Dr returns/contra, Cr **stored-value liability**. Revenue is NOT recognized here.

### `stored_value.redeemed` — NEW
- **Producer:** POS path where a gift card / store credit is tendered against a sale (a tender method on `payment.received`, mirrored as a stored-value movement).
- **Future consumer:** **P5 Accounting** (draw down the liability; revenue recognized via the linked `sale.created`), P12.
- **Payload:** `{ storedValueId, storedValueAccountId, movementId, kind, saleId, paymentId, tenderId, amountMinor, currency, scale, functionalCurrency, functionalScale, fxRateToFunctional, amountFunctionalMinor, redeemedBy, occurredAt }`.
- **Required IDs:** storedValueId, storedValueAccountId, movementId, saleId, paymentId/tenderId (links the redemption to the sale's tender). **Functional twin reserved-nullable** (mirrors `stored_value.issued`) so the liability drawdown posts in functional currency without an FX re-read.
- **GL posting it drives (P5):** Dr **stored-value liability**, Cr cash-clearing (the redemption settles the sale's tender). Balance never goes negative (audited).

> **Exchange flow (Codex gate MEDIUM-6):** an exchange is **NOT a distinct event** — it **decomposes into a linked `sale.refunded` + `sale.created`** that share a common **`exchangeGroupId`** (added to both payloads when `saleType` originates from an exchange; reserved nullable otherwise). This keeps the GL postings symmetric (reverse the returned lines, post the new lines) and orderable (the refund and sale carry stable `originalSaleId`/`saleId`); P5 needs no exchange-specific posting logic. If a future need arises for atomic net-difference accounting, `sale.exchanged` can be added additively — but the decomposition is the locked Phase-4 contract.

## What this map locks for Phase 5 (the consumer)

1. **Every posting field travels on the event** — `cogsMinor`, `taxBreakdown`, `tenders[]`/`fxRateToSale`, `overShort` — so the GL posts deterministically from the event without re-reading mutable OLTP (a precondition for idempotent replay, INV-P5-7).
2. **Stable source + reversal IDs** (`saleId`, `originalSaleId`, `paymentId`, `shiftId`) — the ordering/parking + dedup keys (INV-P5-8).
3. **Reserved-nullable** for what Phase 4 can't populate (`salesRepId`, `customerId`, `restockLocationId`, and the **functional-currency context/twins**) — present-but-null, additive later.
4. **Money is always the triple** (`*Minor` + `currency` + `scale`); FX is carried at tender granularity **AND** every posting amount carries a **functional-currency twin** (`*FunctionalMinor`) + the server-stamped `fxRateToFunctional` — so the GL balances and posts **in functional currency** without an FX-table re-read (INV-P5-4 + INV-P5-7). Reserved-nullable; the rate is captured **at transaction time** (never re-resolved), so reversals reuse the original's rate and replay is deterministic. `sale.voided` is exempt (parks on + reverses the original journal). **Consumer-completeness gate (2026-06-24): the 7 financial events now post every balanced journal from the payload alone.**

> When Phase 4 is built, each emitted payload is locked by a contract test asserting the keys are PRESENT (the Phase-2 `toHaveProperty` discipline), so a refactor can't silently drop a field the GL consumer binds.
