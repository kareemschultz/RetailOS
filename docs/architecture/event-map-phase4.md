# Phase 4 ‚Äî Domain Event Map (POS / Payments / Shifts / Offline)

- **Status:** PLAN / contract doc ‚Äî **no code**. Purpose: lock event **payload shapes + required IDs now**, explicitly **shaped for the Phase-5 Accounting consumer** (the Phase-5 Codex review proved that POS events must carry COGS/tax/tender detail + stable reversal IDs, or the GL cannot post). Same discipline as `event-map-phase2.md`/`event-map-phase3.md`. This is the top Phase-4 readiness blocker (Task C) ‚Äî it must be locked before POS implementation.
- **Transport:** transactional outbox (charter ¬ß24) ‚Äî every event row written in the SAME tenant tx as the mutation (`emitEvent`). `occurredAt` is **server-injected** (last in the spread; device clocks untrusted, ¬ß14). Money = integer `*Minor` + `currency` + `scale`; quantities are **base-unit int8** (Decision #3).
- **Envelope (every event):** `id, type, version (default 1), tenant_id, correlation_id, request_id, payload (jsonb), status (pending), created_at`. Tenant-scoped, correlation/request-ID aware, versioned, replay-safe. **Version/additive-field/upcasting/replay policy: see `event-versioning.md`** (additive fields don't bump `version`; breaking changes bump + add a consumer upcaster).
- **`DomainEventType` constants to ADD (Phase 4):** `sale.refunded`, `sale.voided`, `payment.received`, `shift.opened`, `shift.closed`, `stored_value.issued`, `stored_value.redeemed`. **`sale.created` already exists but is MINIMAL** (`{ saleId, number, totalMinor, currency }` ‚Äî verified) and **must be EXTENDED** to the shape below. Do NOT enumerate an event nothing emits (Phase-2 lesson).

> **Consumer-idempotency contract (Phase-5 INV-P5-7, CRITICAL).** The outbox can REDELIVER. The accounting consumer dedups on **`(tenant_id, outbox_event_id, posting_kind)`** (UNIQUE), posting in the same tx as the journal; a replay returns the existing journal, never re-posts. Therefore every event below carries the data needed to post **deterministically from the event alone** (no re-reading mutable OLTP state at post time).

> **Ordering/parking contract (Phase-5 INV-P5-8).** Reversal events (`sale.refunded`/`sale.voided`) carry `originalSaleId`; if the consumer sees a reversal before its original sale is posted, it **parks** the event until the dependency posts. Hence the stable source/original IDs on every event.

> **Reserved-field discipline (Phase-2 lesson):** any field a later consumer will bind but Phase 4 can't yet populate is reserved **present-but-null**, never shipped absent. Money fields are `_minor`-suffixed; carry `currency` + `scale`.

> **Functional-currency contract (Phase-5 INV-P5-4 ‚Äî consumer-completeness gate, 2026-06-24).** The GL keeps its books in ONE **functional currency** (tenant/company); every journal line must balance **in functional currency** (INV-P5-1) and the consumer must post **without re-reading a mutable FX-rate table** (INV-P5-7). The `tenders[]` carry only `fxRateToSale` (tender‚Üísale) ‚Äî **NOT** the sale‚Üífunctional rate the books need; a redelivered event re-reading a since-corrected rate table would post a different functional amount, breaking deterministic replay. Therefore **every financial event RESERVES (nullable) a functional-currency context** ‚Äî `functionalCurrency`, `functionalScale`, and the **server-stamped `fxRateToFunctional`** (document-currency‚Üífunctional, captured in the SAME tx at transaction time ‚Äî same discipline as server-injected `occurredAt` and the `costing_method_applied` stamp; **never re-resolved later**) ‚Äî **plus a pre-computed `*FunctionalMinor` twin beside every posting amount**, so the journal balances deterministically to the cent without the consumer re-dividing (multi-leg rounding would otherwise force a plug). **Reserved-nullable today; populated when the Phase-5 functional-currency model is locked AND the tenant enables multi-currency.** Single-currency tenants (the Guyana zero-config default): functional == transaction, rate = identity, twins == the `*Minor` amounts ‚Äî **zero setup burden**. **Sequencing constraint (`cross-phase-dependencies.md`):** a historical FX rate cannot be backfilled, so a tenant must not run **multi-currency** POS in production until the producer stamps `fxRateToFunctional` (which needs only a `functional_currency` setting + spot-rate capture ‚Äî a small seam, NOT the full Phase-5 revaluation model). Single-currency POS is unaffected. **`sale.voided` is EXEMPT** ‚Äî it parks on and reverses the original posted journal, carrying no amounts.

## Structural posting invariants ‚Äî the organizing principle (consumer-completeness gate 4, 2026-06-24)

The four review gates found, in order: **missing fields ‚Üí broken symmetry ‚Üí missing multi-currency representation ‚Üí a missing posting dimension (commission)**. The last two are the same failure in different clothes: the contract was built by enumerating the journals we *happened to think of*, then checking those fields were present ‚Äî so each gate "discovered" one more gap (commission was even *referenced* ‚Äî "commission clawback" ‚Äî but carried no amount field). To stop **discovering** gaps and start **designing**, the contract is now governed by two **checkable** invariants, applied UNIFORMLY to every financial event:

- **POST-1 ‚Äî money-representation completeness.** Every monetary quantity on a financial event is the full **quintuple**: transaction `*Minor` + `currency` + `scale` + `*FunctionalMinor` + the **FX rate used** (server-stamped at txn time, never re-resolved). Functional fields reserved-nullable until the P5 functional-currency model lands. *(The gate-3 fix, generalized into a rule.)*
- **POST-2 ‚Äî posting completeness.** For each event, the contract **explicitly enumerates every journal the Phase-5 GL posts** (the full debit/credit set, derived from charter ¬ß19/¬ß20 ‚Äî NOT from memory), and **every amount in every enumerated journal-line travels on the event** (under POST-1). The event is posting-complete **iff** the enumeration ‚Üî payload fields are in 1:1 correspondence. A posting dimension that exists in the charter but carries no field is an INCOMPLETE contract ‚Äî this is exactly how commission was missed.

### Per-event journal enumeration (the POST-2 checklist ‚Äî re-derived, not patched)

| Event | Journals the GL posts (charter ¬ß19/¬ß20) | Amount fields that carry them |
|---|---|---|
| `sale.created` | Dr tender-clearing ¬∑ Cr revenue (subtotal‚àídiscount) ¬∑ Cr VAT-payable (per rate) ¬∑ (Dr COGS, Cr inventory-asset, per line) ¬∑ **(Dr commission-expense, Cr commission-payable ‚Äî if `at_sale`)** | `tenders[].settled*`, `subtotal/discountMinor`, `taxBreakdown[]`, `lines[].cogs*`, **`lines[].lineCommission*` + header `commission*`** |
| `sale.refunded` | reverse revenue/VAT/COGS proportionally ¬∑ restock inventory-asset ¬∑ **reverse commission accrual (clawback)** | (mirror of sale) + **`lines[].lineCommissionClawback*` + header `commissionClawback*`** |
| `sale.voided` | full reversal of the original posted journal (incl. its commission accrual) | parks on `originalSaleId` ‚Äî **no amounts needed** |
| `payment.received` | Dr cash/bank ¬∑ Cr clearing or AR ¬∑ realized FX gain/loss ¬∑ **(commission accrual ‚Äî if `at_settlement`)** | `amount*`, `settled*`, `realizedFxGainLoss*`, **`commission*`** |
| `shift.opened` | Dr drawer-cash ¬∑ Cr float-source (per currency) | `openingFloat[]` |
| `shift.closed` | drawer settlement ¬∑ over/short shrinkage ¬∑ cash movements | `countedCash[]`/`expectedCash[]`/`overShort[]`/`cashMovements[]` |
| `stored_value.issued` | Dr cash-clearing or contra ¬∑ Cr stored-value liability | `amount*` |
| `stored_value.redeemed` | Dr stored-value liability ¬∑ Cr cash-clearing | `amount*` |

**Dimensions enumerated and CLASSIFIED ‚Äî so the checklist is provably complete, not just "commission added":** *discount* posts net (revenue = subtotal ‚àí discount; a gross-revenue + contra-discount policy is a tenant option, and `discountMinor` is already on the event so EITHER policy posts from the payload); *transaction-currency rounding residual* is verifiable from present fields (`subtotal ‚àí discount + tax == total == Œ£tenders`); *loyalty accrual* is **Phase-7 CRM scope**, explicitly out of Phase 4 (reserved there, not here); *tips/gratuity* are restaurant scope, not retail. **After adding commission, the enumeration is complete: every Phase-4 posting dimension either carries its amount (under POST-1) or is justified-derivable / out-of-scope above.**

> **Commission accrual (charter ¬ß19, Decision #7 `commission_accrual_policy`).** Commission is computed from **MUTABLE rules** (flat/%/product/category/rep/tiered) ‚Äî so, like the FX rate and `costing_method_applied`, it must be **resolved at the accrual trigger and STAMPED on the event** (re-resolving at post time could differ from what was owed, breaking replay). The accrual **trigger is policy-dependent** (Decision #7): `at_sale` ‚Üí accrue on `sale.created`, claw back on `sale.refunded` (proportional) / `sale.voided` (100%, via the original-journal reversal); `at_settlement` ‚Üí accrue on `payment.received` (proportional to settled), reverse on cancellation. Both `sale.created` and `payment.received` therefore RESERVE commission fields (nullable); the stamped `commissionAccrualPolicy` tells the consumer which is the live accrual. The refund derives its clawback from the **original sale line's stamped commission** (via `originalSaleLineId`), never re-resolves ‚Äî symmetric and replay-safe (the exact discipline used for `restockedValueMinor`). Commission rides the **sale's** currency + `fxRateToFunctional` (the payable's FX-at-payout is a later AP concern).

### `sale.refunded` ‚Äî NEW
- **Producer:** `pos.refund` router (first-class return, `saleType="return"`, links `originalSaleId`).
- **Future consumer:** **P5 Accounting** (reverse revenue/tax/COGS proportionally), commission clawback (INV-P4-7/Decision #7), P12.
- **Payload:** `{ saleId, originalSaleId, number, locationId, companyId, shiftId, currency, scale, functionalCurrency, functionalScale, fxRateToFunctional, saleType: "return", exchangeGroupId, subtotalMinor, discountMinor, taxMinor, totalMinor, subtotalFunctionalMinor, discountFunctionalMinor, taxFunctionalMinor, totalFunctionalMinor, commissionAccrualPolicy, commissionClawbackMinor, commissionClawbackFunctionalMinor, lines:[{ saleLineId, originalSaleLineId, skuId, productId, lotId, qtyBase, unitPriceMinor, lineTaxMinor, taxRateId, restockedValueMinor, cogsCurrency, cogsScale, restockedValueFunctionalMinor, costingMethodApplied, restockLocationId, lineCommissionClawbackMinor, lineCommissionClawbackFunctionalMinor }], taxBreakdown:[{ taxRateId, rate, baseMinor, taxMinor, baseFunctionalMinor, taxFunctionalMinor }], tenders:[{ tenderId, method, tenderCurrency, tenderScale, amountMinor, fxRateToSale, settledAmountMinor, settledFunctionalMinor }], refundReason, refundedBy, occurredAt }`. **`exchangeGroupId` reserved nullable** ‚Äî set (shared with the paired `sale.created`) only when this return is the inbound leg of an exchange; null otherwise (re-gate MEDIUM). **Functional-currency fields reserved-nullable** per the contract above; the reversal must reverse **exactly** the original posting's functional amounts ‚Äî to keep replay deterministic and symmetric, the refund's `fxRateToFunctional` / `*FunctionalMinor` are taken from the **original sale's stamped rate** (via `originalSaleId`/`originalSaleLineId`), NOT re-quoted at refund time (a re-quote would post a spurious FX gain/loss on a pure reversal).
- **Required IDs:** saleId (the return doc), **`originalSaleId` (ordering/parking key)**, per-line `originalSaleLineId`. **`restockedValueMinor` (full triple with `cogsCurrency`/`cogsScale`) = the value RESTOCKED to inventory.** ‚öÝÔ∏è **Producer note (Codex gate HIGH-4):** `applyValuation()` returns `cogsMinor: 0` on a positive/restock movement (COGS is computed on *issue*, not receipt), so the producer CANNOT take this from the costing primitive on the restock leg. The Phase-4 return path **derives `restockedValueMinor` from the original sale line's posted COGS basis** (referenced via `originalSaleLineId` ‚Äî the original `sale.created` stamped per-line `cogsMinor`+`costingMethodApplied`), so the refund reverses **exactly** what the sale posted (symmetric inventory-asset/COGS reversal). `costingMethodApplied` carries the original method. `restockLocationId` reserved nullable (damaged-return ‚áí may not restock ‚Äî then `restockedValueMinor=0`). `taxBreakdown` is the locked shape (matches `sale.created`) for per-rate VAT reversal. `tenders[]` carry full FX (match sale/payment) for realized-FX on the refund tender. Partial refunds carry only the refunded lines/qty.
- **GL posting it drives (P5):** reverse the proportional sale postings (revenue/VAT per `taxBreakdown`); restock inventory-asset at `restockedValueMinor`, reversing COGS; **reverse commission accrual ‚Äî Dr commission-payable, Cr commission-expense per `commissionClawbackMinor` (when `at_sale`; derived from the original line's stamped `lineCommissionMinor`, proportional to refunded qty ‚Äî never re-resolved)**.
- **Commission clawback note:** `commissionClawbackMinor`/`lineCommissionClawbackMinor` (+ functional twins) are **derived from the ORIGINAL sale line's stamped commission** (via `originalSaleLineId`), proportional to the refunded quantity ‚Äî the same replay-safe derivation used for `restockedValueMinor` (never re-resolve mutable commission rules at refund time). Null under `at_settlement` (commission accrued/reversed on the payment leg, not the sale).

- **Producer:** POS path that issues a gift card or store credit (sale of a gift card, or store credit granted on a return). Writes the stored-value balance ledger (INV-P4-5).

### `stored_value.redeemed` ‚Äî NEW
- **Producer:** POS path where a gift card / store credit is tendered against a sale (a tender method on `payment.received`, mirrored as a stored-value movement).
- **Future consumer:** **P5 Accounting** (draw down the liability; revenue recognized via the linked `sale.created`), P12.
- **Payload:** `{ storedValueId, storedValueAccountId, movementId, kind, saleId, paymentId, tenderId, amountMinor, currency, scale, functionalCurrency, functionalScale, fxRateToFunctional, amountFunctionalMinor, redeemedBy, occurredAt }`.
- **Required IDs:** storedValueId, storedValueAccountId, movementId, saleId, paymentId/tenderId (links the redemption to the sale's tender). **Functional twin reserved-nullable** (mirrors `stored_value.issued`) so the liability drawdown posts in functional currency without an FX re-read.
- **GL posting it drives (P5):** Dr **stored-value liability**, Cr cash-clearing (the redemption settles the sale's tender). Balance never goes negative (audited).

> **Exchange flow (Codex gate MEDIUM-6):** an exchange is **NOT a distinct event** ‚Äî it **decomposes into a linked `sale.refunded` + `sale.created`** that share a common **`exchangeGroupId`** (added to both payloads when `saleType` originates from an exchange; reserved nullable otherwise). This keeps the GL postings symmetric (reverse the returned lines, post the new lines) and orderable (the refund and sale carry stable `originalSaleId`/`saleId`); P5 needs no exchange-specific posting logic. If a future need arises for atomic net-difference accounting, `sale.exchanged` can be added additively ‚Äî but the decomposition is the locked Phase-4 contract.

## What this map locks for Phase 5 (the consumer)

1. **Posting completeness (POST-2):** every event **enumerates every journal the GL posts** (see the checklist above) and **every enumerated journal-line amount travels on the event** ‚Äî `cogsMinor`, `taxBreakdown`, `tenders[]`, `overShort`, **and now `commission*` (the dimension gate 4 surfaced)** ‚Äî so the GL posts deterministically without re-reading mutable OLTP/rules (a precondition for idempotent replay, INV-P5-7).
2. **Stable source + reversal IDs** (`saleId`, `originalSaleId`, `paymentId`, `shiftId`, per-line `originalSaleLineId`) ‚Äî the ordering/parking + dedup keys (INV-P5-8); reversals/clawbacks derive from the original's stamped values, never re-resolve.
3. **Reserved-nullable** for what Phase 4 can't populate (`salesRepId`, `customerId`, `restockLocationId`, the **functional-currency context/twins**, and the **commission fields**) ‚Äî present-but-null, additive later.
4. **Money is always the quintuple (POST-1):** `*Minor` + `currency` + `scale` + `*FunctionalMinor` + the server-stamped FX rate used ‚Äî so the GL balances and posts **in functional currency** without an FX-table re-read (INV-P5-4 + INV-P5-7). The rate (and commission, and costing method) is captured/stamped **at transaction time** (never re-resolved), so reversals reuse the original's stamp and replay is deterministic. `sale.voided` is exempt (parks on + reverses the original journal). **Gate 4 (2026-06-24): the contract is governed by POST-1/POST-2 as checkable invariants; the per-event enumeration is provably complete (commission folded; discount/rounding derivable; loyalty = P7).**

> When Phase 4 is built, each emitted payload is locked by a contract test asserting the keys are PRESENT (the Phase-2 `toHaveProperty` discipline), so a refactor can't silently drop a field the GL consumer binds.
