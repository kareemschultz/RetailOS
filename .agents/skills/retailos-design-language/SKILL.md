# RetailOS Design Language (v1 — governs UI generation, Phase 4+)

## Status & how to use this
This is the **VISUAL SOURCE OF TRUTH** for every UI surface generated by any tool (Claude Design, v0, shadcn Studio, Magic UI, Figma MCP). It is NOT used during backend phases (1–3, 5–7, 9–10, 13). When a UI-bearing phase begins, this skill is law.

- **FIRM now (won't change):** the color system, typography rules, and the semantic principles below.
- **RESERVED (deliberately loose until real screens exist):** exact component pixel-specs and per-module layouts. Don't over-specify these before building — implementation reshapes specifics. Where a number appears in a RESERVED section, treat it as a starting target to tune against real screens, not a hard contract.

Reference baseline screenshots and their extracted principles live in `docs/architecture/ui-inventory/design-references.md`.

## Reference, not law
The owner's dashboard reference screenshots define the **TABLE/ADMIN aesthetic** — clean Notion/Airtable-style data tables, semantic status chips, subdued secondary text, in-card sparklines. RetailOS is table-heavy, so these are the baseline for admin/catalog/inventory **table** surfaces. They are **NOT** the law for POS, warehouse, or dashboard surfaces — those follow their module rules below. RetailOS is an offline-first ERP, not a database tool: never let "match the screenshot" override "would a cashier understand this in a dim room, one-handed."

## Brand shift
The reference screenshots use a green status accent; **RetailOS uses BLUE as its primary accent** (active nav, primary buttons, selected tabs, charts, links, focus). Green/amber/red remain **semantic only** (success/warning/error). Blue is for emphasis, never everywhere — over-using it flattens the hierarchy it's meant to create.

## Aspires to / never
- **Aspires to:** Linear, Stripe Dashboard, Vercel, Notion, Retool, Airtable, modern Odoo — enterprise ERP density + modern SaaS polish + calm, zero clutter.
- **Never:** SAP GUI, legacy Dynamics, old Odoo themes, Bootstrap/AdminLTE admin templates, generic Tailwind CRUD.

## Color
- Primary accent: RetailOS Blue (emphasis only).
- Light: bg white / surface `#FAFAFA` / card white / subtle gray borders.
- Dark: bg `#0A0F1A` / surface `#101827` / card `#111827` / border `#1F2937`. In dark mode, signal depth with **lighter card shades, NOT drop shadows** — shadows read poorly on dim POS screens.
- Status (semantic, always a chip — never raw text like "ACTIVE"): Active green · Inactive gray · Pending amber · Failed red · Draft blue-gray. Always icon + text (Lucide/Phosphor), **never color-only** — colorblind users and glare-washed screens need the redundancy.

## Typography & spacing
- Single sans (Inter or Geist). Headers: tighten letter-spacing ~-2%, line-height ~110% — this is what makes dense SaaS read premium.
- Financial/quantity numbers: **monospace, tabular numerals, RIGHT-aligned** so place values stack and the eye can scan a column.
- **Spacing: a strict 4-point grid** (4, 8, 12, 16, 24, 32px). Consistent spacing on a fixed scale is what lets dense data breathe instead of feeling cramped or arbitrary.

## Radius (RESERVED — starting targets, tune in real screens)
Page cards ~20–24px · dialogs/sheets ~20px · tables ~16px · inputs/buttons ~12px · badges pill · avatars full. Avoid `rounded-none`/`rounded-sm` unless density genuinely demands it.

## Layout, hierarchy & invisible UI
- Every screen flows: **Header → Context → Actions → Metrics → Main work area → Secondary insights.** Never header-then-100-buttons-then-table.
- Priority order: user task → key metrics → main workflow → secondary actions → advanced settings. Advanced/destructive actions live behind kebab menus, drawers, sheets — never on the primary path.
- **Invisible UI:** truncated text, icon-only buttons, and ambiguous labels must carry hover tooltips. The absence of this "invisible UI" is the tell of a beginner dashboard — and on a warehouse floor an unlabeled icon is a guess, not an action.

## Tables (the heart of RetailOS)
Sticky headers, virtualization, column resize/reorder, saved views, bulk actions, export, search, faceted filters, keyboard nav, and **density toggles** (Comfortable / Compact / Dense — the same table serves a touch POS and a dense accounting screen).

## ERP is workflow-driven, not CRUD-driven
Not "Create Product / Save" — a wizard: General → Pricing → Inventory → Suppliers → Images → Review. Long forms = wizard, never one giant page. All forms: autosave drafts, field help, validation, undo, a review step.
- **Interaction states:** every button/input has 5 explicitly designed states — default, hover, active (pressed), disabled, loading. A control with no pressed/loading state leaves the user unsure whether their tap registered, which on a slow-network POS causes double-submits.

## Progressive disclosure (role-aware)
A warehouse worker sees Receive / Move / Count / Ship — NOT valuation layers, landed cost, FIFO consumption (manager-tier). Reveal complexity gradually; the right default view is the one that matches the role's job.

## Multi-view per module (a real edge over Odoo/NetSuite/Dynamics)
Where applicable each module supports **Table · Card · Detail · Timeline** — and where it fits, **Calendar · Kanban · Map · Hierarchy**.
- **Warehousing gets a HIERARCHY view:** Warehouse → Zone → Aisle → Rack → Bin (ties directly to the Phase-3 self-referential location tree).
- **Audit trails & stock-movement history: TIMELINE first** (Receipt → Transfer → Adjusted → Sold → Returned), table second — a movement history reads as a story, never as a giant raw table.

## Offline UX (first-class — this is an offline-first ERP)
Sync state is **always visible, never hidden**: Online (green) · Syncing (blue) · Queued (amber) · Conflict/Failed (red) — icon + text, not color-only. A cashier mid-outage must know at a glance whether a sale is safe.

## Dashboards
Answer three questions: what needs attention? what changed? what should I do next? — exceptions / KPIs / insights, NOT 50 random charts. Charts sparingly: line / bar / area / donut. No radar / 3D / pie-explosions.

## Motion
Purposeful only — page/drawer transitions, toast, number ticker, chart entry. **Forbidden:** floating particles, bouncing buttons, spinning dashboards, animated tables — **especially on POS**, where motion is latency the cashier feels. Honor `prefers-reduced-motion`.

## Accessibility
WCAG 2.2 AA — keyboard nav, visible focus rings, screen-reader labels, high contrast, large touch targets.

## Module device targets
- **POS:** touch-first, large controls, near-zero animation, always-visible totals + sync state.
- **Warehousing:** scanner-first, big task buttons, tablet.
- **Inventory:** table + timeline + location hierarchy.
- **Accounting:** spreadsheet-dense, monospace, right-aligned.
- **CRM:** relationship/timeline-driven.
- **Executive:** KPIs, insights, exceptions, forecasts — not data dumps.

## Global search (mandatory)
Command-K / Raycast-feel — one place searches products, customers, suppliers, invoices, transfers, locations, employees, reports.

## Empty states
Never "No Data" — explain what goes here, show the next step, provide a CTA.

## Golden rule (the governor over everything, including the screenshots)
Would a cashier understand this? Would a warehouse worker use it one-handed? Would a CFO trust this? Would a CEO get it in 5 seconds? If not — simplify.
